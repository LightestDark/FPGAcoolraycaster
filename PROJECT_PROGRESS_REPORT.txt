2) File-by-File Explanation
============================================================
Note:
- The standalone source files below were consolidated into [raycaster.v](raycaster.v) to keep the project compact.
- They are listed here as module descriptions, not as separate current source files.

A) vga_sync.v
Purpose:
- Generates VGA horizontal/vertical timing counters and sync pulses for 640x480@60Hz.

Inputs:
- clk_25: pixel clock (~25 MHz).
- rst: reset.

Outputs:
- hsync: active-low horizontal sync.
- vsync: active-low vertical sync.
- hc: horizontal pixel counter (0..799).
- vc: vertical line counter (0..524).
- active: high only in visible region (hc<640 and vc<480).

Internal behavior:
- Counters increment each pixel clock.
- Horizontal wraps every 800 clocks.
- Vertical increments when horizontal wraps.
- Sync windows:
  - HS low from hc=656 to 751
  - VS low from vc=490 to 491

Dependencies:
- None (standalone timing module).


B) tb_vga_sync.v
Purpose:
- Testbench for vga_sync.v.

Behavior:
- Generates a 25 MHz clock (40 ns period).
- Asserts reset briefly.
- Dumps waveforms to vga_sync.vcd.
- Runs enough time for 3 full scan lines.

Dependencies:
- Instantiates vga_sync.

Generated artifact:
- vga_sync.vcd


C) sin_cos_lut.v
Purpose:
- 256-entry angle lookup table for sine and cosine.

Interface:
- Input: angle[7:0] (0..255 maps to 0..360 degrees).
- Outputs: sin_out and cos_out in signed Q1.14 scale (~ +/-16384).

Internal behavior:
- Tables are initialized with $sin/$cos and $rtoi in an initial block.
- Combinational lookup returns values for current angle.

Dependencies:
- None.

Usage notes:
- Good for direction vectors in a raycaster (fixed-point friendly).


D) map_rom.v
Purpose:
- Stores 16x16 maze map where 1=wall, 0=floor.

Interface:
- Inputs: tile_x[3:0], tile_y[3:0]
- Output: wall

Internal behavior:
- 16 rows of 16-bit constants.
- wall = map[tile_y][tile_x]

Dependencies:
- None.

Usage notes:
- Shared map definition used by tb_lut_map and conceptually by ray traversal.


E) tb_lut_map.v
Purpose:
- Combined testbench for sin_cos_lut and map_rom.

Behavior:
- Tests sin/cos at angles 0,64,128,192.
- Tests map samples at (0,0), (15,15), (1,1).
- Dumps lut_map.vcd and prints values.

Dependencies:
- Instantiates sin_cos_lut and map_rom.

Generated artifacts:
- lut_map.vcd
- lut_map.vvp (compiled simulation executable)


F) ray_dda.v
Purpose:
- First-pass ray march core to find first wall hit and an approximate ray distance in steps.

Interface:
Inputs:
- ray_x_q8_8, ray_y_q8_8: ray origin in Q8.8 fixed-point tile units.
- dir_x_q1_14, dir_y_q1_14: direction vector in Q1.14 fixed-point.
Outputs:
- hit: 1 if collision found.
- hit_tile_x, hit_tile_y: tile coordinates of first hit.
- dist_steps: step count to hit.
- side: rough indicator of last major stepping axis.

Internal behavior:
- Uses 1/16 tile per step (STEP_FRAC=16 in Q8.8).
- Marches up to MAX_STEPS=320.
- Tests map occupancy via internal is_wall() function (same map constants embedded).
- Stops at first wall or out-of-bounds.

Dependencies:
- Standalone for now (does not instantiate map_rom directly).

Design note:
- This is a simulation-friendly stepping core, not yet a cycle-accurate pipelined hardware engine.


G) tb_ray_dda.v
Purpose:
- Testbench for ray_dda.

Behavior:
- Sets player origin at tile (1.5,1.5) in Q8.8.
- Casts 3 rays: East, South, North-West.
- Prints hit tile and distance steps.
- Dumps ray_dda.vcd.

Dependencies:
- Instantiates ray_dda.

Generated artifacts:
- ray_dda.vcd
- ray_dda.vvp


H) write_files.py
Purpose:
- Temporary helper script used during file recovery to rewrite sources.

Current relevance:
- Not required for normal build/sim flow.
- Can be kept for reference or removed for cleanliness.

Update:
- Removed to keep the source tree concise.


I) vga_raycast_demo.v
Purpose:
- Integrated visual demo module that combines VGA timing, angle lookup, ray hit estimation, movement, and wall shading.

Interface:
- Inputs: clk_25, rst, move_fwd, move_back, turn_left, turn_right, strafe_left, strafe_right
- Outputs: hsync, vsync, vga_r[3:0], vga_g[3:0], vga_b[3:0]

Internal behavior:
- Uses vga_sync counters to derive a ray angle per screen column.
- Uses sin_cos_lut for direction vectors.
- Uses ray_dda to estimate wall hit distance.
- Updates player position and angle once per frame from movement controls.
- Converts distance to wall slice height and applies side-based shading, checker texture, and distance fog.
- Draws sky and floor gradients outside wall slice region.

Dependencies:
- vga_sync.v
- sin_cos_lut.v
- ray_dda.v


J) tb_vga_raycast_demo.v
Purpose:
- Testbench for the integrated VGA raycast demo.

Behavior:
- Generates clock/reset.
- Drives movement controls to verify player motion and rotation.
- Dumps vga_raycast_demo.vcd.
- Runs long enough to cover multiple frames and capture motion.
- Prints sampled RGB values for selected center-line columns.

Dependencies:
- vga_raycast_demo.v


K) tb_vga_judge_demo.v
Purpose:
- Judge-facing proof test that validates autonomous movement and visible scene change without manual inputs.

Behavior:
- Leaves all movement inputs at 0.
- Confirms player position and camera angle change over time (auto-demo mode).
- Confirms center-line RGB samples vary while the camera moves.
- Dumps vga_judge_demo.vcd.

6) Debugging Timeline and What Was Fixed
============================================================
Problem 1:
- Icarus reported: "No top level modules, and no -s option."
Root cause:
- Some .v files accidentally contained shell heredoc command text instead of Verilog source.
Fix:
- Replaced corrupted file contents in sin_cos_lut.v, map_rom.v, tb_lut_map.v with valid Verilog modules.

Problem 2:
- Ray DDA initially returned immediate hit at distance 0.
Root cause:
- Ray origin was placed inside a wall tile.
Fix:
- Updated testbench player origin to empty tile (1.5,1.5).

Problem 3:
- Some rays returned no hit.
Root cause:
- MAX_STEPS too small for long traversals across map.
Fix:
- Increased MAX_STEPS from 96 to 320.

Result:
- All current simulations now compile and produce expected outputs.
- The project now includes a first complete integrated rendering path in simulation.

7) Engineering Approach Used
============================================================
Approach summary:
1. Stabilize base video timing first (vga_sync).
2. Validate math/data building blocks independently (sin_cos_lut, map_rom).
3. Implement and verify ray collision core in isolation (ray_dda).
4. Use testbenches and VCD traces at each stage before integration.

Why this approach:
- Reduces integration risk.
- Makes bugs local and easier to diagnose.
- Produces report-friendly evidence for each subsystem.

Verification method:
- For timing logic: inspect waveforms in GTKWave.
- For LUT/ROM/data path: compare printed numeric checkpoints to expected values.
- For traversal logic: verify hit tiles and distance progression from known positions.

Refinement pass method:
- Converted testbenches to pass/fail style with explicit checks.
- Fixed timing check alignment for registered sync outputs.
- Tightened types (signed LUT outputs) and cleaned comments.
- Re-ran full regression after each change.
